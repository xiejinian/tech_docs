00 · V2.0 核心更新

本版本两项核心优化

基于资深呼叫中心技术专家评审意见，在 V1.0 方案基础上新增以下两项关键改进。

🔇
优化一：FreeSWITCH/ASR 双层降噪
V1.0 仅在 ASR 侧做单层降噪，噪音在 VAD 检测前未经处理，影响 VAD 精度。V2.0 新增 FreeSWITCH 侧 Speex NS 作为第一层，专攻稳态噪音；ASR 侧 RNNoise 作为第二层，处理残留非稳态噪音。两层分工明确，整体降噪效果大幅提升。
⚡
优化二：编码帧直接 Tap 获取核心码流
V1.0 通过 RTP 旁路监听获取音频，需经过 FreeSWITCH 抖动缓冲层（20–60ms 延迟）。V2.0 新增自定义 mod_raw_tap 模块，在抖动缓冲前直接 Tap 编码帧，通过 Unix Socket 推送给 ASR，净节省 15–40ms 延迟，使端到端响应更接近人耳舒适阈值。
版本说明：本文档涵盖 V1.0 全部内容，所有 V2.0 新增内容以 NEW 标注，更新内容以 UPD 标注，其余章节与 V1.0 一致。
01 · Executive Summary

方案 概述

在不重构核心架构的前提下，为现有 FreeSWITCH 呼叫中心系统增加自然、低延迟的智能打断能力。

🎯
核心目标
在 TTS 播报过程中实时判断用户是否具备"有效发言意图"，在 110ms 内完成打断决策并切换至 ASR 识别流程，显著提升交互自然度与用户满意度。
🏗️
方案核心
在 ASR 模块内集成打断决策引擎（IDE），融合 VAD、声学特征、语义意图、对话上下文四维信号加权评分，统一对外发布打断事件。
📐
架构原则
ASR 模块为打断信号的唯一权威来源，TTS 模块与业务层保持被动执行者角色，各模块职责清晰、可独立迭代升级，避免跨模块状态耦合。
📈
预期收益
打断准确率目标 ≥ 90%，误触发率 ≤ 5%，端到端延迟优化至 ~110ms，通话自然度评分提升 30%+，支持差异化业务场景动态配置。
关键设计决策：打断的本质是"用户输入信号"与"系统输出行为"的仲裁。ASR 模块负责裁决"用户想说话了吗"，业务层负责策略决定"现在该不该让他说"，TTS 模块负责执行停播。三层职责清晰分离，是本方案最核心的架构决策。

03 · Dual-Layer Noise Suppression V2 NEW

FreeSWITCH / ASR 双层降噪架构

V1.0 仅在 ASR 侧降噪，噪音已影响上游 VAD 判断。V2.0 引入两层分工明确的降噪机制，在声音进入 VAD 之前完成双重清洗。

第一层 · FreeSWITCH
Speex NS 稳态降噪
实现方式
Speex 编解码器内置 NS，修改一行配置即可启用，零额外部署成本
处理目标
稳态噪音：底噪、线路噪音、空调声、稳定的背景环境音
处理方向
仅处理上行（用户侧音频），不影响下行 TTS 播报质量
延迟
< 5ms，对通话质量无感知影响
在编码帧 Tap 点之后、送入 Unix Socket 之前完成
算法简单高效，适合大量并发通话场景
显著减轻第二层神经网络模型的处理压力
→
第二层 · ASR 模块
RNNoise 非稳态精降噪
实现方式
RNNoise 神经网络降噪模型，在 PCM 解码后、VAD 检测前运行
处理目标
非稳态残留噪音：偶发杂音、混响、叠加人声、第一层未能消除的成分
处理位置
位于解码器之后、VAD 检测之前，确保 VAD 收到最干净的音频
延迟
< 20ms，神经网络推理，精度远高于传统算法
保障 VAD 和声学特征分析的输入质量
因第一层已预处理，计算量显著降低
可选备选：NVIDIA NsNet2（需 GPU）
关键设计原则：两层降噪的价值不是"降噪两次"，而是分工不同、各司其职。第一层在 FreeSWITCH 侧用极低代价消除稳态噪音，第二层在 ASR 侧用高精度神经网络处理复杂残留噪音。组合效果远优于任何单层方案，且两层均只处理上行用户侧音频，完全不影响 TTS 下行播报质量。
04 · Raw Frame Tap V2 NEW

编码帧直接 Tap 绕过抖动缓冲

V1.0 通过 RTP 旁路监听获取音频，必须等待抖动缓冲完成（20–60ms）。V2.0 在缓冲之前直接 Tap 编码帧，大幅压缩延迟。

❌ V1.0 旧路径
1.RTP 数据包到达 FreeSWITCH
2.进入抖动缓冲（Jitter Buffer）等待 20–60ms
3.抖动缓冲输出，开始解码为 PCM
4.RTP 旁路嗅探，复制 PCM 传给 IDE
5.IDE 开始 VAD + 分析
总延迟：~140ms（含抖动缓冲 20–60ms）
✅ V2.0 新路径（mod_raw_tap）
1.RTP 数据包到达 FreeSWITCH
2.mod_raw_tap 在抖动缓冲前 Hook 编码帧
3.编码帧经 Unix Socket 直推 ASR 进程
4.ASR 自解码 G.711/G.729 为 PCM
5.RNNoise 降噪 → VAD → 分析
总延迟：~110ms（节省 25–40ms）

关于 ASR 新增解码职责： 编码帧 Tap 方案要求 ASR 模块自行处理 G.711（μ-law / a-law）解码，这是额外新增的职责。 G.711 解码算法极为简单（查表法，纯 CPU，延迟约 1ms），不会成为性能瓶颈， 且相关开源库（如 libg711）已非常成熟。 G.729 解码需要额外引入 bcg729 库，若环境中使用 G.729 需确认授权。 实际节省的 20–50ms 抖动缓冲延迟远大于解码新增的 ~1ms，净收益显著。

06 · Scoring Model

四维加权评分模型

多维信号融合输出 0–1 打断置信度，超过阈值则触发打断。双层降噪显著提升各维度输入质量。

打断分数 = 0.30 × VAD分 + 0.25 × 声学特征分 + 0.30 × 语义意图分 + 0.15 × 上下文分
综合得分 ≥ 0.65 触发打断 · 精确关键词命中则直接进入快速通道
Dimension 01
VAD 语音活动检测
WebRTC VAD 一级过滤，延迟 <1ms，作为信号门控层
Silero VAD 二级确认，精度更高，缓存 100ms 后判断
V2：双层降噪后 VAD 输入更干净，误触发率显著降低
最低有效语音持续时长 120ms，过滤短暂噪音
Dimension 02
声学特征分析
HNR（谐波噪声比）：人声 >10dB，噪音 <5dB
基频 F0：人声集中 80–800Hz，超范围则降分
V2：降噪后 HNR 提取精度提升，人声/噪音区分更准确
频谱质心：语音集中在 300–3000Hz 带内
Dimension 03
语义意图识别
流式 ASR 实时转写，不等完整句子即获取部分结果
关键词规则库优先匹配（<1ms）：停止 / 确认 / 异议类
V2：更低延迟的音频送达，ASR 转写更及时，语义更准确
本地轻量意图分类模型兜底，约 10MB，延迟 <50ms
Dimension 04
对话上下文评分
TTS 播报超 3 秒：用户更可能想打断，动态加分
当前句为长句（>15s）：适当降低打断阈值
用户历史打断率高：倾向于认为是有效打断
近期发生误打断：触发冷却，提升本次阈值

V2.0 新增两项风险：mod_raw_tap 模块稳定性、G.729 授权问题。

风险项等级影响描述应对措施
mod_raw_tap 模块稳定性 NEW中自研 FS 模块可能引入崩溃或内存泄露，影响通话质量充分单元测试 + 灰度上线 + 保留旁路 RTP 作为降级兜底方案
G.729 codec 授权 NEW中若环境使用 G.729，bcg729 解码库存在专利授权风险提前确认编解码器清单，G.729 场景评估替换为 G.711 或 Opus
高噪音环境误触发低→低V2 双层降噪后此风险显著下降，但极端噪音仍可能触发SNR 自适应阈值 + HNR 声学过滤（已因降噪质量提升而更准确）
语义模型延迟超标中推理延迟超 80ms 导致端到端超 110ms本地 GPU 部署 + 关键词快速通道绕过模型推理
预缓冲音频截头中打断决策时用户已说 200ms，首部语音丢失300ms 滚动预缓冲，打断时将缓冲音频回传 ASR
