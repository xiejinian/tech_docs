需求理解确认

我们如何理解这次招标的真实目标

表面任务

招募 C1 方法论顾问、C2 平台工程顾问、C3 流程组织顾问，外加 E1 落地专家，在 6 个月内交付 15 项文档/平台/培训成果（D01–D15），按人天/里程碑结算。

真实目标

把"AI 写代码"从少数极客的个人习惯，变成全公司研发流程里默认存在、可度量、可考核的一环——这是一次工程治理 + 组织变革项目，不是一次工具采购。

通读交付件清单后，我们注意到一个关键信号：15 项交付件里，纯"平台/技术"类（D03–D06）只占 4 项，而"方法论规范类"（D01-D02/D07-D09）与"组织落地类"（D10-D15）合计 11 项。这说明贵司真正的瓶颈不是缺工具，而是缺一套让 AI 辅助编码可控、可信、可推广的工程方法与组织机制。我们的方案因此把 C1、C3 放在与 C2 同等甚至更靠前的优先级上。

"AI Coding 能力建设"项目最常见的失败模式不是技术选型错误，而是平台上线后没人用、用了没人信、信了也没法规模复制。本方案的每一项交付，都直接对应贵司在『5.验收标准』里设定的四个硬指标——AI 占比、交付周期、试点数量、认证人数。

C1  C3 方法论 / 流程组织顾问

飞凡科技主导。核心顾问拥有阿里巴巴/蚂蚁集团多条产品线研发负责人经历，长期参与阿里交易技术委员会、钉钉技术委员会工作，并主导过蘑菇街电商中台建设——这类经验的共同点是：在大规模研发组织里，把"个体最佳实践"沉淀为"全员可复制的流程规范"。这正是 C1（Spec-Driven 方法论、工作流状态机）与 C3（角色重塑、认证体系、组织文化变革）两个方向所需要的核心能力。

03

方法论与技术路线

这是方案的核心——按 C1 / C2 / C3 三个方向展开具体落地路径

C1 方法论体系：从"个人会用 AI"到"团队有章法"

C1 的核心产出是 Spec-Driven Development（规范驱动开发） 与 Harness Engineering（AI 生成代码的安全护栏工程）。前者解决"AI 该基于什么生成代码"，后者解决"AI 生成的代码该如何被信任"。两者共同支撑工作流状态机（D02）的设计：每一次 AI 辅助编码，都从"需求转规范"开始，经过明确状态流转，而不是程序员直接对着对话框写 prompt。

四层防线模型（D01/D02 核心内容）：

1
规范层
需求转化为结构化 Spec 文档（接口定义、验收条件、边界场景），作为 AI 生成代码的唯一输入源，杜绝"自然语言随意描述"导致的歧义。
↓
2
生成层
Agent 按 Spec 生成代码、测试用例与开发日志（dev-log），生成过程全程留痕，纳入度量看板统计 AI 代码占比。
↓
3
验证层
CI/CD 质量门禁自动校验（单测覆盖率、静态扫描、Spec 一致性比对），未通过自动打回生成层，不进入人工环节浪费评审资源。
↓
4
人工复核层
人工只复核"业务逻辑是否正确"与"架构是否合理"，不再做逐行代码审查——这是交付周期能下降 40% 的关键设计，而非简单地让 AI 多写代码。

角色契约（人机分工边界）

明确定义"AI 可独立决策的范围""AI 必须征求人工确认的范围""人工必须亲自完成的范围"三类边界，写入团队规范文档，避免"AI 写的代码出了问题该谁负责"这类组织摩擦反复发生。

工作流状态机设计

需求 → 规范确认 → AI生成 → 自动验证 → 人工复核 → 合并，六个状态节点，每个节点有明确的进入/退出条件与责任人，可直接映射进现有的工单系统或 CI 流程。

C2 平台与知识工程：五大能力的建设顺序

D04 提到的 Rules / Skills / MCP / Agent / Hook 五大平台能力，常见误区是同时铺开建设，导致资源分散、谁都不成熟。我们建议的建设顺序遵循"先约束、再扩展"原则：

顺序能力为什么先做这个典型周期
①Rules定义代码规范、安全红线、禁止模式——是后续所有能力的"宪法"，必须最先确定，否则 Agent/Skills 无据可依W2–W4
②Hook在 CI/CD 关键节点插入自动校验钩子，把 Rules 转化为可执行的强制检查，而非贴在墙上的文档W3–W6
③Skills把团队已验证有效的代码模式、领域知识封装成可复用单元，是知识沉淀的最小颗粒度W4–W9
④MCP连接内部系统（代码仓、工单、知识库），让 Agent 具备"读取真实上下文"的能力，避免凭空生成W4–W10
⑤Agent前四项就位后再编排自动化程度更高的 Agent 工作流，此时风险最低、可控性最高W6–W12

知识工程底座架构（D05）

内部资产市场（沉淀本企业的 Skills/Rules/MCP 配置，可被各团队复用、检索、二次组合）+ 通用知识库（接入代码仓与历史文档做 RAG 检索）。两者解耦设计：前者是"团队的最佳实践库"，后者是"企业的知识记忆体"，避免混为一谈导致检索精度下降。

度量看板与 Token 成本治理（D06）

三类核心指标：①生成层指标（AI 代码占比、采纳率）②验证层指标（自动校验通过率、人工复核耗时对比）③成本指标（按团队/项目维度的 Token 消耗与单位代码成本），三者打通才能向管理层证明"降本"而非仅"增效"。

C3 流程组织变革：把方法论"种"进组织里

C1 解决"怎么做"，C2 解决"用什么做"，C3 解决最难的部分——"为什么大家要改变现在的做法"。这是大多数 AI Coding 项目最终流于形式的真正原因。

传统→Spec-Driven 流程映射（D07）

逐条对照现有 SDLC 节点（需求评审/技术方案/编码/测试/上线）与新流程节点的映射关系，让团队看到"不是推翻重来，是在现有流程里插入新环节"，降低变革阻力。

角色重塑与能力模型（D08）

定义"AI 协作工程师"等新角色画像与能力分级标准，配套认证体系（对应验收指标"认证≥40人"），让转型有清晰的个人成长路径，而非单纯的考核压力。

AI-First 文化与 Hackathon（D09）

W18–W24 安排团队级 Hackathon，用真实业务场景而非演示场景检验平台能力，同时是最有效的内部口碑传播机制——这比任何培训 PPT 都更能说服怀疑者。

验收指标对齐说明

四个硬指标，逐项说明方案如何支撑达成

AI 占比 ≥ 70%

四层防线模型把"生成层"明确为 AI 主导环节，人工只做业务逻辑复核（第4层），从机制上而非口号上保证 AI 承担大部分编码工作量。D06 看板从 W8 起持续采集真实数据，给管理层提供过程可见性，而非到结项才知道是否达标。

交付周期 ↓ 40%

提速来源不是"AI写得快"，而是验证层自动校验取代了大部分人工逐行审查（详见第3节四层防线说明）。这意味着即使 AI 占比未达 70%，交付周期也能因评审环节精简而下降——两个指标互为支撑而非简单线性绑定。

试点 ≥ 6 个验收通过

9 个试点（D11）覆盖在 W7-W20，预留约 3 个试点的失败/延期容错空间。试点选型标准在 Phase 0 阶段与贵司联合确定，优先选择"业务边界清晰、历史代码可作为对照基准"的场景，降低验收争议。

认证 ≥ 40 人

D08 能力模型与 D14 培训课件共用同一套标准，培训全程同步而非集中在项目末期突击进行，给受训人员留出实际应用、辅导、补考的时间窗口，避免"为了凑数仓促发证"。

分级风险识别与应对建议
🔴 严重风险

试点选择标准缺失若试点业务场景选择不当（边界模糊、依赖外部系统过多），会直接拖累 D11/D13 的验收质量与案例可信度。建议：Phase 0 阶段（W1-4）必须产出书面试点筛选标准并经贵司业务方确认，不可凭经验口头拍板。

🔴 严重风险

组织阻力导致"平台建成、无人使用"这是 AI Coding 类项目最常见的失败模式。建议：D09 Hackathon 不安排在项目末期走形式，而是在试点已有真实成果后举办，用内部口碑而非强制要求驱动采用率。

🟡 高风险

平台与企业现有工具链冲突D03 架构选型阶段若未充分调研现有 CI/CD、代码仓、工单系统的接口能力，后续 Hook/MCP 建设可能受阻。建议：架构评审报告（D03）必须包含与现有系统的兼容性专项章节，而非仅做技术选型对比。

🟡 高风险

人天投入与驻场强度不匹配JD 要求"前3个月连续投入、后3个月按需驻场"，若试点数量/复杂度超出预期，E1 团队投入强度可能不足。建议：第6节人天估算预留弹性区间，并在 Phase 1 结束（W14）时做一次中期资源校准。

🔵 中风险

知识库内容质量参差D05 通用知识库若接入的历史文档/代码仓质量不一，会拉低 RAG 检索准确率，间接影响 Agent 生成质量。建议：知识接入前做一轮数据质量筛查，而非全量直接灌入。

🔵 中风险

认证体系与实际晋升机制脱节若 D08 认证标准只是"考核"而不与岗位晋升、绩效挂钩，会削弱团队参与积极性，影响认证人数达标。建议：认证体系设计阶段即与贵司 HR/人才发展部门同步对接。
