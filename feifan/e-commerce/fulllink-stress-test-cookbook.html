1
压测体系总览与核心原则
全链路压测不是简单地打流量，而是一套闭环体系
阿里实践参考 · 技术峰会公开分享

阿里巴巴全链路压测（Full-Link Stress Testing）始于 2014 年双11备战，核心思路是：在生产环境中使用真实流量路径，通过"影子"机制隔离测试数据，确保压测结果真实反映系统容量，避免测试环境与生产环境的差异导致误判。

1.1 全链路压测 vs 传统压测

对比维度传统压测全链路压测
测试环境独立测试环境生产环境（影子隔离）
流量路径单接口/单服务完整业务链路（浏览→加购→下单→支付）
测试数据测试库/Mock数据影子库（基于生产数据副本）
结果真实性低（环境差异大）高（反映真实容量）
实施成本低高（需要改造各层中间件）
风险无线上风险需要严格的隔离机制

1.2 核心原则

原则一
生产可见
压测必须在生产环境路径上跑，测试环境结论不可信
原则二
数据隔离
压测产生的所有数据必须与真实数据完全隔离
原则三
流量真实
按真实业务比例模拟流量，不能只压单一接口
原则四
随时可停
压测必须有一键停止机制，发现异常立即中断
原则五
问题必改
压测发现的问题不修复不得进入大促，不能带病上阵
原则六
逐步加压
从目标容量的 20% 开始，梯度加压，观察系统行为
2
影子库/影子表方案设计
数据隔离是全链路压测的基础工程，做不好即是灾难

2.1 影子数据库方案对比

方案实现方式优点缺点推荐场景
影子库（推荐） 独立数据库实例，库名加 _shadow 后缀 完全隔离，风险最低 成本较高，需额外资源 核心链路（订单/支付）
影子表 同库不同表，表名加 _shadow 后缀 成本低，实施快 共享DB资源，可能影响生产 非核心链路（日志/行为）
影子Schema 同实例不同Schema 中等成本 仍共享数据库连接池 中小团队妥协方案

染色（Traffic Coloring）是全链路压测的灵魂：在请求入口注入压测标识，该标识在整个调用链中透传，每个服务节点依据标识决定路由到影子资源。

Java · 压测标识透传（ThreadLocal + Filter）
// 1. 定义压测上下文
public class StressTestContext {
    private static final ThreadLocal<Boolean> STRESS_FLAG =
        new InheritableThreadLocal<>();

    public static void mark() { STRESS_FLAG.set(true); }
    public static boolean isStressTest() {
        return Boolean.TRUE.equals(STRESS_FLAG.get());
    }
    public static void clear() { STRESS_FLAG.remove(); }
}

// 2. Zuul Filter 识别并透传压测标识
@Component
public class StressTestFilter extends ZuulFilter {
    @Override
    public Object run() {
        RequestContext ctx = RequestContext.getCurrentContext();
        HttpServletRequest req = ctx.getRequest();

        String stressHeader = req.getHeader("X-Stress-Test");
        if ("true".equals(stressHeader)) {
            StressTestContext.mark();
            // 向下游服务透传Header
            ctx.addZuulRequestHeader("X-Stress-Test", "true");
        }
        return null;
    }
}

// 3. MyBatis 拦截器 — 自动替换为影子表
@Intercepts({@Signature(type=StatementHandler.class,
             method="prepare", args={Connection.class, Integer.class})})
public class ShadowTableInterceptor implements Interceptor {
    @Override
    public Object intercept(Invocation invocation) throws Throwable {
        if (StressTestContext.isStressTest()) {
            StatementHandler handler = (StatementHandler) invocation.getTarget();
            MetaObject meta = SystemMetaObject.forObject(handler);
            String sql = (String) meta.getValue("delegate.boundSql.sql");
            // 将表名替换为影子表：orders → orders_shadow
            String shadowSql = replaceTablesToShadow(sql);
            meta.setValue("delegate.boundSql.sql", shadowSql);
        }
        return invocation.proceed();
    }
}

中间件影子化方案关键配置
MySQL 影子库（独立实例）或影子表（表名后缀） ShardingSphere 动态数据源 + MyBatis 拦截器

🚨 绝对禁止：压测期间向第三方支付（支付宝/微信支付）、短信服务、物流系统发送真实请求。必须在压测环境中对这些外部调用做 Mock 拦截，否则会产生真实交易和费用。

压测模型与容量规划
科学估算峰值，设计真实的压测场景

3.1 峰值流量估算公式

阿里实践参考 · 容量规划方法论

阿里容量规划的核心公式：目标 TPS = 日峰值订单量 × 流量放大系数 / 业务链路中最慢节点 RT。大促场景通常以日常峰值的 3~5 倍作为压测目标，并预留 20% Buffer。

容量规划计算示例
【业务背景】
日常峰值 TPS：500（每秒500笔交易）
大促目标放大倍数：3x（达到大促 3 倍峰值）
压测目标 TPS：500 × 3 = 1500 TPS
安全Buffer 20%：1500 × 1.2 = 1800 TPS（压测实际打到此值）

【接口 TPS 分解（根据业务漏斗比例）】
商品详情页：   1800 × 10  = 18000 QPS  (每笔订单10次浏览)
加购接口：      1800 × 3   =  5400 QPS  (每笔订单3次加购)
下单接口：      1800 × 1   =  1800 TPS  (基准)
支付接口：      1800 × 0.7 =  1260 TPS  (70%下单后支付)
查询订单状态：  1800 × 5   =  9000 QPS  (支付后频繁轮询)

【Little公式：并发数估算】
并发数 = TPS × RT(秒)
示例：下单接口 RT = 200ms → 并发数 = 1800 × 0.2 = 360 个并发

3.3 压测数据准备要求

📦 测试账号准备
  • 压测专用账号：≥ 并发数 × 10 个
  • 账号需有历史订单数据（模拟真实用户）
  • 账号余额充足（影子支付可通过）
  • 账号状态正常，非黑名单
🛍️ 商品数据准备
  • 影子库商品数量：≥ 生产商品量 1/10
  • 压测商品库存充足，不限库存（避免因缺货中断压测）
  • 商品含真实图片、描述、SKU（避免特殊逻辑绕过）
  • 覆盖不同价格段、分类、促销类型
💡 压测执行建议：推荐使用 JMeter（开源）或阿里云 PTS（商业）。建议先在凌晨低峰期做摸底压测（50% 目标量），验证影子方案无误后，再在业务低峰期（早7-9点）执行正式压测，避免对生产造成影响。

5.1 接口 RT 飙升排查链路

1
定位慢接口：查 APM 调用链（SkyWalking/Zipkin）

找到耗时 Top5 的接口，查看调用链中哪个节点耗时最长（DB？Redis？下游服务？）

2
若是 DB 慢：查慢查询日志 + EXPLAIN

参考《SQL优化手册》第1章诊断流程，重点看是否有全表扫描、filesort、锁等待。

3
若是 Redis 慢：查 slowlog + 热 Key 分析
Redis CLI
# 查看慢命令
SLOWLOG GET 20

# 热Key分析（redis-cli --hotkeys 需Redis 4.0+）
redis-cli --hotkeys -a password

# 查看大Key（可能导致阻塞）
redis-cli --bigkeys
4
若是线程池耗尽：用 Arthas 实时诊断
Arthas · 实时诊断
# 查看线程状态，找BLOCKED/WAITING线程
thread -n 20

# 定位线程在哪里阻塞（如锁等待）
thread -b

# 监控某方法执行时间（找慢方法）
trace com.example.OrderService createOrder -n 5

# 实时监控方法调用统计
monitor com.example.OrderService createOrder -c 5

限流降级预案
压测验证容量上限，预案保障大促安全兜底

6.1 Sentinel 热点参数限流配置

Java · Sentinel 热点限流
// 热点商品防刷：对同一 skuId 限频
@SentinelResource(
    value = "getProductDetail",
    blockHandler = "productDetailFallback"
)
public ProductDTO getProductDetail(Long skuId) {
    // 业务逻辑...
}

// Sentinel 控制台配置（等价的代码配置）
ParamFlowRule rule = new ParamFlowRule("getProductDetail")
    .setParamIdx(0)           // 第0个参数（skuId）
    .setGrade(RuleConstant.FLOW_GRADE_QPS)
    .setCount(100)            // 每个skuId每秒最多100次
    .setDurationInSec(1);

// 特例：秒杀商品额外限制更严
ParamFlowItem specialItem = new ParamFlowItem()
    .setObject(String.valueOf(seckillSkuId))
    .setClassType(String.class.getName())
    .setCount(10);           // 秒杀商品限制到10QPS
rule.setParamFlowItemList(Collections.singletonList(specialItem));
ParamFlowRuleManager.loadRules(Collections.singletonList(rule));

6.2 Hystrix 熔断阈值参考配置

⚠️ 关键约束：Hystrix timeout 必须 > Ribbon ConnectTimeout + ReadTimeout（含重试次数）。公式：HystrixTimeout > (ConnectTimeout + ReadTimeout) × (1 + MaxAutoRetries) × (1 + MaxAutoRetriesNextServer)
YAML · application.yml 熔断配置
# Ribbon超时（基础配置）
ribbon:
  ConnectTimeout: 1000        # 连接超时1s
  ReadTimeout: 3000           # 读取超时3s
  MaxAutoRetries: 0           # 同实例重试0次（大促推荐）
  MaxAutoRetriesNextServer: 1 # 切换下一实例1次

# Hystrix超时：必须 > Ribbon总时间
# = (1000+3000) × (1+0) × (1+1) = 8000ms，留buffer设10000ms
hystrix:
  command:
    default:
      execution.isolation.thread.timeoutInMilliseconds: 10000

    # 下单接口单独配置（更严格的SLA）
    OrderService#createOrder:
      execution.isolation.thread.timeoutInMilliseconds: 5000
      circuitBreaker:
        requestVolumeThreshold: 20      # 触发熔断的最小请求数
        errorThresholdPercentage: 50    # 错误率超50%熔断
        sleepWindowInMilliseconds: 5000 # 熔断5s后尝试恢复

6.3 三级降级预案

降级级别触发条件降级动作用户感知
L1 自动降级 Hystrix 自动熔断 返回缓存数据或默认值，不调用下游 展示缓存商品信息，功能基本正常
L2 功能降级 压测/大促达到 80% 容量 关闭非核心功能（推荐、评论、历史足迹） 部分功能不可用，核心下单链路正常
L3 流量管控 系统濒临崩溃 排队页 / 限流弹窗 / 只放核心用户 看到排队提示，体验降级但不报错

避坑清单
全链路压测的常见翻车场景（来自真实复盘）
  • 影子方案未覆盖 MQ 消费者：压测消息流入生产消费者，导致影子订单触发真实物流或短信通知。必须确保影子 Topic 有对应的影子消费者部署。
  • 定时任务在压测期间触发：对账、推送等定时任务恰好在压测时间窗口执行，严重干扰压测结果，必须在压测前暂停所有非关键定时任务。
  • 压测数据污染生产数据库：影子表名配置错误，数据写入了生产表。每次压测前必须验证：向影子库写入一条测试记录，确认生产库中无对应记录。
  • 只压单接口得出错误结论：单独压下单接口可能轻松达标，但真实大促时商品详情页的并发会同时打满 DB，必须按业务漏斗比例同时施压。
  • 压测流量全来自同一 IP：被接入层（Nginx/WAF）误识别为攻击流量限速，导致压测结果失真。压测工具需配置分布式出口 IP。
  • 忽略 JVM 预热阶段：压测开始的 1~2 分钟内，JIT 未充分编译，RT 偏高，该阶段数据不应纳入结果统计。建议先低流量预热5分钟。
  • Hystrix 超时 < Ribbon 超时：导致请求在 Ribbon 还在等待时就被 Hystrix 熔断，触发大量不必要的熔断，掩盖了真实瓶颈。
  • 压测完不清理影子数据：影子库数据持续积累，下次压测时"历史数据"影响扫描行数，进而影响索引选择，导致前后压测结果不可比较。
  • 监控没有提前验证：压测期间发现监控面板无数据（采集配置问题），无法判断系统状态，盲目继续加压。监控可用性必须在压测前做端到端验证。
  • 人工宣布"达标"没有量化标准：必须预先定义明确的达标标准（成功率 / P99 / 资源水位），避免主观判断导致带着隐患上大促。
