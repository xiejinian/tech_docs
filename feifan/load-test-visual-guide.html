标准执行流程（6 步）

01
需求提交D-5研发/业务负责人
填写压测申请单，明确业务背景、接口清单、峰值预期、数据准备需求、Owner。技术负责人 24 小时内确认排期。
02
方案设计评审D-4 ~ D-3压测工程师
输出流量模型、数据模型、监控方案、风险与降级预案、明确通过标准。经技术负责人评审后方可执行。
03
数据与环境准备D-2 ~ D-1研发 + DBA
造测试数据（用户/商品/库存/券），缓存预热，监控告警就绪，PTS 脚本验证通过，降级开关就绪。
04
压测执行D Day 21:00压测工程师
预热 → 稳态（30min）→ 梯度拉升 → 峰值冲刺（150%）→ 故障注入（L3必做）→ 恢复观测。
05
结果分析D Day压测工程师 + 研发
分析接口/资源/DB/缓存/MQ 各维度指标，标注 P0/P1/P2 问题，每个问题明确负责人和截止时间。
06
整改与复测闭环D+1 ~ D+3各问题负责人
P0：24h 内整改复测。P1：发布前复测通过。P2：记录技术债，下迭代跟进。结论写入复盘报告归档。

压测执行阶段（D Day）

推荐时间窗口：工作日 21:00 ~ 23:00，避开业务高峰

阶段时长流量观测重点
预热 5 min 10% 目标 QPS 无异常则进入稳态
稳态 20~30 min 80% 目标 QPS 每 5 分钟记录 RT / 错误率 / 资源
梯度拉升 10 min 每 2min +10%，到 120% 识别性能拐点，确认限流触发点
峰值冲刺 5 min 150% 目标 QPS 验证限流/熔断/降级全部生效
故障注入 L3 必做 Redis 延迟 / 慢 SQL / MQ 停消费者 观察超时行为与补偿机制
恢复观测 10 min 流量停止后 MQ 清零、库存对账、系统自恢复
! 立即停止压测条件：核心接口错误率 > 5% 持续 1 分钟 / 数据库连接池耗尽 / 出现脏数据（库存负数、重复下单）/ 主机 CPU 持续 > 90% 超过 2 分钟。

重试策略规范

规范项要求
允许重试的接口仅幂等接口（GET / 带幂等 Key 的 POST）
最大重试次数1 ~ 2 次
重试间隔指数退避：100ms → 200ms → 400ms
禁止重试下单、扣库存、支付（非幂等操作，重试会产生重复数据）
禁止行为多层链路同时重试（会将流量放大 N 倍）
! 压测时重试策略必须与生产完全一致，不允许"压测特供配置"，否则结果无参考价值。

全链路压测技术保障方案

以下 8 项技术能力是全链路压测能够落地、结果可信的前提条件。每项均需输出对应的方案文档并通过评审，缺少任何一项都会导致压测效果大打折扣或产生线上风险。

! 技术保障方案必须在压测执行 D-5 前完成评审，D-2 前完成验证，否则本次压测应延期。
T1 流量染色与链路隔离方案 最高优先级，其他方案的前置
为什么需要
不隔离压测流量会污染线上数据，造成虚假订单、库存错乱、用户数据脏写。
核心内容
· 请求头统一打标 X-Pts-Test: true
· 网关/Filter 识别标识，写入 ThreadLocal / MDC
· 所有下游调用透传标识（HTTP Header / MQ 属性）
· 日志中自动附加 [PTS] 前缀以便过滤
就绪判断
✓ 压测请求在 SLS 可被独立过滤
✓ 标识跨 MQ 消费者正确透传
✓ 未染色请求不进入压测路径
! 若无法实现完整染色，禁止在生产环境执行全链路压测，只能压测环境（隔离机房）压测。
T2 影子库 / 影子表方案 压测写操作的数据隔离核心
为什么需要
压测产生的订单、支付、库存扣减等写操作不能进入生产表，否则需要大量人工清理且存在资金风险。
核心内容
· 方案 A（推荐）：影子表（同库前缀 pts_），MyBatis Plugin 动态路由
· 方案 B：独立影子数据库，连接串动态切换
· Redis：影子 Key 前缀 pts:{原key}
· MQ：独立 Topic 或 消息 Tag 过滤
· 第三方（支付/短信）：沙箱/Mock 替换
就绪判断
✓ 压测订单写入 pts_hic_orders
✓ 库存扣减写影子 Redis Key
✓ 生产表记录数压测前后无变化
✓ 支付回调指向 Mock 服务
T3 测试数据构造与清理方案 数据真实性决定压测有效性
为什么需要
数据分布不真实（如只有几个用户/SKU 循环）会导致缓存命中率失真、热点聚集，压测结论无法指导生产。
核心内容
· 用户数据脚本（≥10万，覆盖分销层级）
· 商品 SKU 数据（≥5000，热点/非热点分层）
· 历史订单（≥100万条，保证分页真实）
· 优惠券（每用户 ≥2 张，含过期/已用场景）
· 压测结束后自动清理影子表脚本
就绪判断
✓ 数据构造脚本可幂等执行
✓ 清理脚本经 DBA 评审
✓ 数据规模满足场景需求
✓ 热点 SKU 分布符合业务预期
T4 缓存预热方案 避免冷启动导致数据库被击穿
为什么需要
压测开始时缓存为空，流量直接打到 DB，造成 DB 被击穿，且 RT 曲线前段失真。
核心内容
· 库存 Hash 预写脚本（所有热点 SKU）
· 商品详情缓存预热接口
· 活动/营销配置缓存预热
· 用户 Token 批量预生成
· 预热完成确认脚本（检查 Key 存在性）
就绪判断
✓ Redis 热点 Key 命中率 > 90%
✓ 预热后 DB QPS 基线平稳
✓ 预热脚本执行时间 < 30 分钟
T5 限流 / 熔断 / 降级方案 保护系统不被压垮的最后防线
为什么需要
没有限流/熔断，超量流量会拖垮整个集群，且压测无法验证系统自我保护能力。
核心内容
· Sentinel 规则配置（接口 QPS 上限 / 线程数上限）
· 熔断规则（RT 异常比例阈值）
· 降级开关（商品详情走静态 / 库存跳过校验）
· 开关管理平台（可在压测中实时开关）
· 降级后的用户提示文案
就绪判断
✓ 限流触发后返回友好错误码
✓ 熔断后自动恢复验证通过
✓ 降级开关可在 1 分钟内生效
✓ 限流日志可在 SLS 中查询
T6 幂等性保障方案 防止重试产生重复数据
为什么需要
高并发下网络抖动频繁，PTS 自动重试 + 业务重试叠加，非幂等接口会产生重复订单、重复扣款。
核心内容
· 下单接口唯一幂等 Key（客户端生成 UUID）
· Redis 幂等去重（SETNX + 过期时间）
· MQ 消费者去重（消息 ID 幂等表）
· 支付回调幂等（订单状态机保护）
· 幂等失败返回原成功结果而非错误
就绪判断
✓ 同一幂等 Key 重复调用 10 次，DB 仅有 1 条记录
✓ MQ 重复投递不产生重复消费
✓ 压测结束重复下单率 = 0
T7 全链路可观测性接入方案 没有监控的压测是盲测
为什么需要
压测发现问题的前提是能看到瓶颈所在，没有链路追踪就无法定位是 DB、Redis 还是代码逻辑慢。
核心内容
· ARMS Java Agent 接入（所有服务节点）
· SLS 日志采集 + 压测 Tag 过滤 Dashboard
· PTS 与 ARMS 联动（Trace ID 关联）
· DB 慢 SQL 自动采集（RDS 慢查询日志）
· Redis 热 Key / 大 Key 监控开启
· 关键告警规则提前配置（见第 9 章）
就绪判断
✓ ARMS 拓扑图可见全链路调用
✓ 慢 SQL 可在 ARMS 中定位到具体 XML
✓ SLS Dashboard 压测流量可单独过滤
✓ 告警规则压测前 D-2 验证触发正常
T8 压测应急预案与快速止损方案 出现问题时能在 1 分钟内止损
为什么需要
压测可能触发未知 Bug 或超预期负载，需要能在极短时间内中止影响并恢复系统。
核心内容
· PTS 一键停止压测入口（所有人知晓）
· 全局降级开关（一键开启所有降级）
· 库存回滚脚本（影子 Redis Key 清理）
· 影子表快速清理脚本（DBA 预演过）
· 应急联系人清单（研发/DBA/运维/业务）
· 立即停止压测的判断标准（见第 4 章）
就绪判断
✓ 应急预案文档已共享给所有参与人
✓ 全局降级开关验证 < 60s 生效
✓ 清理脚本在测试环境预演通过
✓ 应急联系人均在线（压测当晚）
