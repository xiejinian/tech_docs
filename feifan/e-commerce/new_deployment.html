01
背景与目标
📋
现状与问题

客户现有架构采用 APIsix → Nginx → Zuul 1.x 三层串联网关，服务间通信方式未统一，存量组件中包含多个已停止维护的 Spring Cloud Netflix 组件（Ribbon、Hystrix、Zuul）。三层网关叠加导致链路延迟增加、运维复杂度翻倍、故障排查困难。

⚠️

迁移策略：逻辑迁移，非代码自动转换。存量业务代码按功能模块逐步迁移，新框架作为目标态，业务逻辑平移进来，不做机械的 1:1 代码替换。

三个核心目标

网关从三层精简为一层 服务间 RPC 统一为 Dubbo 3.x 服务治理统一到 SCA（Nacos + Sentinel） 停维组件全部替换 架构精简原则：能少则少

04
Dubbo 协议说明与演进路径
⚡
当前 · 目标协议对比

当前：经典 Dubbo 协议（TCP）

  • 序列化：Hessian2
  • 传输：TCP 长连接，单连接多路复用
  • 注册：Nacos 接口级注册
  • 适用：纯 Java 微服务内网调用
  • 优点：性能最优，Sentinel 集成最简单
  • DTO 要求：实现 Serializable
  • 配置：dubbo.protocol.name=dubbo

未来：Triple 协议（HTTP/2 + Protobuf）

  • 序列化：Protobuf（强类型，体积小30%+）
  • 传输：HTTP/2，天然支持流式调用
  • 跨语言：可被 gRPC 客户端直接调用
  • 接口：从 .proto 文件生成，需重新定义 DTO
  • 切换成本：接口层重写，业务逻辑不变
  • 配置：dubbo.protocol.name=tri

核心配置示例

product-service/application.yml — Dubbo 核心段
dubbo:
  application:
    name: product-service
  protocol:
    name: dubbo           # 切 Triple 改为: tri
    port: -1              # 自动选端口
    serialization: hessian2  # Triple 时改为: protobuf
    threads: 200
  registry:
    address: nacos://${NACOS_ADDR:localhost:8848}
  provider:
    timeout: 3000
    retries: 0            # Provider不重试，避免写接口重复执行
    filter: sentinelProviderFilter
  consumer:
    check: false
    timeout: 3000
    retries: 2            # 读接口可重试
Provider @DubboService · Consumer @DubboReference
// ── Provider（product-service）──────────────────────
@DubboService(version = "1.0.0", group = "ecommerce")
@SentinelResource(value = "ProductService#getById",
    blockHandler = "getByIdBlocked", fallback = "getByIdFallback")
public class ProductServiceImpl implements ProductService { ... }

// ── Consumer（product-facade / order-service）────────
@DubboReference(version = "1.0.0", group = "ecommerce",
    check = false, timeout = 3000,
    retries = 1)   // 写接口设 retries=0
private ProductService productService;

中间件使用覆盖矩阵
🔧
Demo 工程覆盖的所有中间件能力

Dubbo 3.x RPC

  • @DubboService Provider
  • @DubboReference Consumer
  • version/group 版本管理
  • 经典协议 + Triple 注释演进

Nacos

  • 服务注册发现
  • 配置中心（shared-configs）
  • Sentinel 规则动态推送
  • Gateway 动态路由

Sentinel

  • Provider 端 QPS/线程数限流
  • Consumer 端熔断降级
  • Gateway 入口限流
  • 慢调用/异常比例熔断规则

Redis

  • 旁路缓存（Cache-Aside）
  • 批量 multiGet 缓存
  • 分布式锁（Lua原子解锁）
  • MQ消费幂等Key
  • Gateway 令牌桶限流

MySQL + MyBatis-Plus

  • 逻辑删除（@TableLogic）
  • 自动填充（MetaObjectHandler）
  • 分页插件（PaginationInnerInterceptor）
  • JSON字段（JacksonTypeHandler）

Elasticsearch

  • 官方 Java API Client
  • 索引自动初始化（@PostConstruct）
  • IK 中文分词 Mapping
  • 多字段全文搜索（name^2 权重）
  • ES 异常降级到 MySQL

RocketMQ

  • 异步 Producer（asyncSend）
  • Consumer（幂等去重）
  • Topic/Tag/Group 常量管理
  • 失败重试机制
  • ES 索引异步同步

Gateway + JWT + Seata

  • JWT 鉴权全链路
  • 用户上下文 ThreadLocal
  • @GlobalTransactional 框架
  • MapStruct DTO→VO 转换

启动步骤

1

准备环境

前置：Docker Desktop + JDK 21 + Maven 3.9+ · 复制环境变量：cp .env.example .env

2

启动全部中间件 + 业务服务

cd deploy && docker-compose up -d · MySQL 自动执行 01_init.sql 建表并插入测试数据 · healthcheck 保证中间件就绪后服务才启动

3

导入 Sentinel 规则

登录 Nacos 控制台 http://localhost:8848/nacos → 配置管理 → 新建：将 deploy/sentinel/ 下两个 JSON 文件内容分别创建为 product-service-flow-rules 和 product-service-degrade-rules

4

验证服务健康

  • Nacos：http://localhost:8848/nacos（nacos/nacos）· 检查服务列表
  • Sentinel：http://localhost:8858（sentinel/sentinel）
  • ES：http://localhost:9200/ecommerce_product（验证索引自动创建）
  • API：curl http://localhost:80/api/product/1
5

启动前端（可选）

cd frontend && npm install && npm run dev · 访问 http://localhost:5173

⚠️

ES IK 分词器：默认 ES 镜像不含 IK 插件，启动后执行 ElasticsearchConfig 的 mapping 创建会使用 ik_max_word 分词器，若插件未安装会报错。解决方案：① 将 analyzer 临时改为 standard 先跑通；② 或在 docker-compose 中使用预装 IK 的自定义镜像。
