01 — 产品定位

三条产品线 · 分层覆盖不同客户需求

平台不是单一产品，而是三条有机联动的产品线，共用同一个基础设施，面向不同客户场景分别定价。

⚡
核心产品线：AI服务网关 + 记忆引擎
面向所有B端企业客户的基础产品。提供LLM/TTS/ASR多模态服务转发，叠加会话内和跨会话记忆能力，大幅降低客户的Token消耗成本。这是平台的收入主体，也是其他两条产品线的流量入口。
  • 多模态API统一网关（LLM/TTS/ASR）
  • 会话内上下文智能压缩（降低单次Token消耗）
  • 跨会话长期记忆（用户偏好/历史上下文持久化）
  • New API计费体系（余额/账单/额度管理完整托管）
  • 企业多租户管理（子账号/权限/用量统计）
👨‍💻
增值产品线：程序员智能插件（高阶套餐）
面向技术团队和开发者的独立插件产品，以高阶套餐包形式定价。基于红熊记忆科学底座，提供代码学习记忆、编程上下文增强等能力。定位是"让AI更了解你的代码库和编程习惯"，而非取代Cursor。
  • 代码仓库语义理解与记忆（跨项目上下文）
  • 编程习惯和偏好的长期记忆
  • IDE插件集成（VS Code / JetBrains）
  • 团队代码规范和最佳实践的知识沉淀
  • 与核心产品线共用同一计费体系
🏢
企业集成产品线：企业知识库集成
以集成形式提供，不是平台自建知识库，而是帮助企业客户把现有的知识管理系统（Notion、Confluence、SharePoint等）接入平台的记忆引擎，让AI调用企业私有知识。
  • 主流知识库系统的标准化接入适配器
  • 企业私有数据的本地向量化（数据不出企业）
  • 私有知识与公有模型的安全隔离架构
  • 自定义RAG参数调优服务
🎯
竞争定位：我们和谁不同
明确区分竞争赛道，避免定位模糊。
  • vs 纯API代理（302.ai等）：我们不只是转发，记忆引擎是核心增值
  • vs Cursor：我们不做编辑器，做的是跨工具的记忆层
  • vs LangChain Memory：我们是开箱即用的SaaS，不是开发者框架
  • vs 国内AI厂商：我们专注出海合规，对接海外模型能力

04 — 合规框架

合规是产品设计，不是法务后置

合规不能只靠服务协议，需要技术架构和协议条款双重保障。以下是当前判断的核心风险点和应对方案，正式运营前必须有律师介入确认。

⚠️
重要声明：以下合规框架为业务层面的初步设计，不构成法律意见。正式运营前，必须委托具有中新两地执照的涉外律师事务所，对PIPL数据出境合规、新加坡主体实质运营要求、VAS牌照（如适用）等进行专项法律评估并出具书面意见。
🏛️主体架构：新加坡实体要求
新加坡主体不只是注册一个公司，监管机构会看"实质"——是否有真实的本地运营、本地董事、本地办公地点。如果实际控制人和运营团队全在中国，新加坡主体的合规效力会打折扣。
建议：在新加坡设立至少一名本地常驻联系人/董事，并在运营上体现新加坡实质（如银行账户、当地合同等）。
🔒PIPL数据出境：最核心的合规风险
如果企业客户的数据（即使是B端数据）包含中国境内个人信息，通过平台流转到新加坡服务器，可能触发PIPL第38-40条的数据出境要求，包括安全评估、标准合同等。
技术上：平台不存储原始对话内容，仅存储经过脱敏处理的记忆摘要。协议上：明确客户对数据出境合规负主责，平台仅作为数据处理者。
📋客户准入KYC：技术+协议双防线
仅靠服务协议声明"仅限面向海外业务企业"不够，需要在技术和流程上有配套措施，否则一旦有客户将平台用于国内业务，平台可能承担连带责任。
准入流程：要求企业客户提交海外业务证明材料（海外公司注册证明、海外业务合同摘要等），并在平台协议中明确违规使用的违约责任和赔偿条款。
🌐海外API合规：灰色地带的边界管理
Claude/GPT等海外模型在国内使用存在政策不确定性。"仅面向海外业务"的限定是重要的合规边界，但需要在技术上也有体现——比如IP地址检测、使用场景声明等。
从红熊处获取其海外算力渠道的合规材料（服务商资质、合同等），作为平台合规的上游背书。同时在平台协议中明确"海外业务用途"的定义和限制。
技术层面的合规保障措施（协议之外必须落地的）
数据最小化存储
平台不存储原始对话内容，记忆引擎只保存经过脱敏和摘要化的语义信息。日志保留不超过90天，自动清理。
数据归属技术保障
每个企业租户的记忆数据物理隔离存储，支持客户一键导出和删除全部数据，平台无法访问已删除数据。
审计日志完整性
所有API调用、数据访问、配置变更均生成不可篡改的审计日志，可供监管审查，存储在新加坡机房。

各层技术选型与理由

选型原则：生产验证优先、开源可控为主、避免单一云厂商锁定。记忆层基于红熊底座，不从零开发。

API网关层 New API（QuantumNous fork）开源 Go 31k+ stars，生产验证，计费体系完整，Docker一键部署，符合飞凡"托管计费"决策 One API（原版，功能稍少）
接入层/边缘 Cloudflare Workers / Nginx JS 新加坡节点覆盖好，DDoS防护内置，TLS终止，CDN加速，成本低 AWS CloudFront + ALB
记忆引擎底座 红熊记忆科学产品 — 合作方既有产品，飞凡不重复造轮子，基于此做工程化封装和高并发运营 Mem0（开源，4k stars）
向量数据库 Qdrant开源 Rust 新加坡可部署，性能强，支持自托管（数据不出境），Docker镜像成熟，Apache 2.0协议 Weaviate / Pinecone（托管）

08 — 风险管理

关键风险与对策

⚡合规风险（最高优先级）
新加坡主体合规实质性要求、PIPL数据出境、海外API使用政策，任何一个踩雷都可能导致平台被迫停止运营。
合规工作与技术开发并行，不能等产品上线再补。第一阶段必须有律师书面意见，合规评估的结论可能影响产品架构，必须提前知晓。
⚡记忆能力效果未经生产验证
Token节省30–60%是理论预估，实际效果取决于用户使用场景和记忆质量。如果效果不达预期，产品核心卖点就会动摇。
MVP阶段优先验证记忆能力的实际Token节省数据，设计A/B测试（开启/关闭记忆对比），用真实数据替代理论数字，再开始大规模推广。
⚡上游资源依赖风险
红熊提供源码和算力，是平台最重要的上游依赖。如果红熊调整政策、源码授权出现争议，或算力供给不稳定，平台核心能力会受影响。
Token源码授权协议中确保商用权和多资源接入权（已在飞凡×红熊授权协议中体现）；技术上同时接入萌招和小融作为备用资源，不单一依赖红熊算力。
⚡冷启动速度风险
目标客户是"面向海外业务的中国企业"，这个群体的触达路径相对窄，商务关系能带来的种子客户数量有限，冷启动可能比预期慢。
第一批客户优先从红熊、小融、萌招的现有客户里转化，这些客户已有AI使用需求且有信任基础。同时准备英文版官网和文档，面向新加坡本地科技企业开拓第二梯队客户。
飞凡AI服务平台综合方案 · 核心逻辑：以记忆引擎为差异化，以New API为计费基础设施，以新加坡合规部署为市场准入门槛。三阶段路线图中，合规工作必须与技术开发并行，不能后置。MVP优先验证记忆能力的实际Token节省效果，用数据驱动后续投入决策。
