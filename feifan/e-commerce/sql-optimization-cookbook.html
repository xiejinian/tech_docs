慢 SQL 识别与诊断体系
建立可量化、可追踪的慢查询治理闭环

1.1 Slow Query Log 配置

慢查询日志是 SQL 问题的第一现场。大促前务必确认以下配置已开启，阈值建议设为 1s，大促期间可收紧至 500ms。

MySQL · my.cnf
# 开启慢查询日志
slow_query_log = 1
slow_query_log_file = /var/log/mysql/slow.log
long_query_time = 1          # 超过1s记录；大促前改为0.5
log_queries_not_using_indexes = 1   # 未使用索引的查询也记录
log_throttle_queries_not_using_indexes = 10  # 每分钟最多记录10条，防日志膨胀
min_examined_row_limit = 100 # 扫描行数 < 100 的忽略
💡 阿里实践：阿里内部通过 DAS（数据库自治服务）对慢查询实现秒级感知和自动推送，平台团队每日早上9点收到前日 Top20 慢 SQL 报告。中型电商可用 pt-query-digest 工具做替代，每日定时跑分析。

1.2 EXPLAIN 解读标准

拿到慢 SQL 后，第一步是 EXPLAIN。以下是关键字段的判读标准：

字段 危险值（需立即处理） 警告值 正常值
typeALL（全表扫描）index（全索引扫描）ref / eq_ref / const
rows> 100万10万 ~ 100万< 1万
ExtraUsing filesort / Using temporaryUsing whereUsing index（覆盖索引）
keyNULL（无索引）非最优索引命中预期索引
filtered< 10%10% ~ 30%> 50%
⚠️ 重点关注：Using filesort 意味着 MySQL 在内存或磁盘进行了排序，大促高并发下极易成为瓶颈。Using temporary 出现在 GROUP BY 场景，需要立即优化。

索引优化策略
正确的索引是 SQL 优化 80% 的解法

2.1 覆盖索引（Covering Index）

当查询所需字段全部包含在索引中时，MySQL 无需回表，直接从索引返回数据。这是大促场景下最值得挖掘的优化点之一。

❌ 需要回表（慢）
-- 只有 user_id 是索引，
-- status/create_time 需回表
SELECT order_id, status, create_time
FROM orders
WHERE user_id = 10086
ORDER BY create_time DESC;
✅ 覆盖索引（快）
-- 建联合索引后无需回表
ALTER TABLE orders
ADD INDEX idx_uid_time_status
  (user_id, create_time, status);

-- 查询命中覆盖索引
SELECT order_id, status, create_time
FROM orders
WHERE user_id = 10086
ORDER BY create_time DESC;
💡 索引设计原则：SELECT 的字段 + WHERE 的字段 + ORDER BY 的字段，优先考虑能否放入同一个联合索引。字段顺序：等值查询字段靠前，范围查询字段靠后，排序字段最后。

2.2 联合索引最左前缀原则

联合索引 (a, b, c) 只有在以 a 开头的查询中才能生效。以下是常见命中/失效场景：

查询条件是否命中索引 (a,b,c)说明
WHERE a=1✅ 命中最左前缀满足
WHERE a=1 AND b=2✅ 命中前两列命中
WHERE a=1 AND b=2 AND c=3✅ 命中全列命中
WHERE a=1 AND c=3⚠️ 部分命中仅用 a 列，c 列不生效
WHERE b=2 AND c=3❌ 未命中跳过最左列 a
WHERE a LIKE '%xxx'❌ 未命中前缀模糊匹配失效
WHERE a LIKE 'xxx%'✅ 命中后缀模糊可命中

2.3 索引失效的典型场景

以下写法是大促前 SQL Review 的重点扫描对象：

❌ 索引失效写法
-- 1. 对列做函数运算
WHERE DATE(create_time) = '2024-11-11'

-- 2. 隐式类型转换
-- phone是varchar，传入数字
WHERE phone = 13800138000

-- 3. OR条件导致失效
WHERE user_id = 1 OR status = 2

-- 4. NOT IN / NOT EXISTS
WHERE status NOT IN (0, 3)
✅ 正确写法
-- 1. 改为范围查询
WHERE create_time >= '2024-11-11 00:00:00'
  AND create_time < '2024-11-12 00:00:00'

-- 2. 显式类型匹配
WHERE phone = '13800138000'

-- 3. 改为UNION
SELECT * FROM t WHERE user_id=1
UNION ALL
SELECT * FROM t WHERE status=2

-- 4. 改为IN
WHERE status IN (1, 2, 4)

3.1 库存扣减防超卖

阿里实践参考 · 公开技术文章

阿里双11的库存扣减采用"分段库存 + 原子操作"方案：将单一库存字段拆分为多个 bucket，减少行锁争用。对于中型电商，可先实现"乐观锁 CAS + 数量校验"的标准方案。

方案一：UPDATE 原子扣减（推荐基础方案）

SQL · 库存扣减
-- ✅ 原子操作，WHERE条件保证不超卖
-- affected_rows = 0 即为库存不足，业务层判断
UPDATE sku_inventory
SET
  available_stock = available_stock - #{quantity},
  locked_stock    = locked_stock + #{quantity},
  update_time     = NOW()
WHERE
  sku_id = #{skuId}
  AND available_stock >= #{quantity}   -- 核心：库存充足才扣减
  AND is_active = 1;

-- Java层判断
-- if (affectedRows == 0) throw new InsufficientStockException();

方案二：分段库存（高并发进阶）

SQL · 分段库存设计
-- 将100件库存分10个bucket，每bucket 10件
-- 每个请求随机选一个bucket，减少行锁竞争
CREATE TABLE sku_inventory_bucket (
  id         BIGINT PRIMARY KEY,
  sku_id     BIGINT NOT NULL,
  bucket_no  TINYINT NOT NULL,         -- 0~9
  stock      INT NOT NULL DEFAULT 0,
  UNIQUE KEY uk_sku_bucket (sku_id, bucket_no)
);

-- 扣减时随机选bucket
UPDATE sku_inventory_bucket
SET stock = stock - #{quantity}
WHERE sku_id   = #{skuId}
  AND bucket_no = #{randomBucket}  -- MOD(requestId, 10)
  AND stock     >= #{quantity};
⚠️ 禁止写法：先 SELECT 再 UPDATE 的"查后改"模式在高并发下必然超卖，即使加了事务也只是把问题推迟了。Redis 预减库存 + DB 最终确认 是大促的标准方案。

3.2 订单列表深分页优化

用户订单列表 LIMIT offset, size 在 offset 很大时（如第100页）会扫描大量数据，这是大促期间个人中心接口变慢的主因。

❌ 深分页写法（LIMIT大偏移）
-- offset=9900时，扫描9910行丢弃9900行
SELECT order_id, status, amount, create_time
FROM orders
WHERE user_id = #{userId}
ORDER BY create_time DESC
LIMIT 9900, 10;   -- 💀 扫描近万行
✅ 游标分页（推荐）
-- 前端传入上页最后一条的 order_id 和 create_time
SELECT order_id, status, amount, create_time
FROM orders
WHERE user_id = #{userId}
  AND (create_time < #{lastCreateTime}
    OR (create_time = #{lastCreateTime}
       AND order_id < #{lastOrderId}))
ORDER BY create_time DESC, order_id DESC
LIMIT 10;  -- ✅ 永远只扫10行
💡 子查询延迟关联方案：若业务必须支持页码跳转，可用延迟关联：先通过索引子查询捞主键，再 JOIN 回原表取完整字段，避免大量回表。
SQL · 延迟关联
-- 延迟关联：内层只查主键（覆盖索引），外层再回表
SELECT o.*
FROM orders o
INNER JOIN (
  SELECT order_id
  FROM orders
  WHERE user_id = #{userId}
  ORDER BY create_time DESC
  LIMIT 9900, 10         -- 只查主键，成本低
) ids ON o.order_id = ids.order_id
ORDER BY o.create_time DESC;

3.3 热点数据缓存与 DB 一致性

阿里实践参考 · 公开架构文章

阿里双11商品详情页采用多级缓存（CDN → 本地缓存 → Redis → DB），对 DB 的读请求在活动期间降至 1% 以下。数据一致性通过 Binlog + MQ 异步刷新解决，强一致场景才走 DB。

缓存穿透、击穿、雪崩 SQL 层防护

问题现象SQL/DB层应对
缓存穿透 大量查询不存在的 key，直打 DB DB查询结果为空也缓存空值（TTL 30s），布隆过滤器前置拦截
缓存击穿 热点 key 过期瞬间大量请求涌入 DB 分布式锁（Redisson）保证只有一个请求回查 DB，其余等待
缓存雪崩 大量 key 同时过期，DB 瞬间压力飙升 TTL 加随机抖动（base + random(600s)），DB 连接池限流兜底

4.1 ShardingSphere-JDBC 路由规则

ShardingSphere-JDBC 通过 SQL 语义自动识别读写，默认规则如下：

SQL 类型默认路由说明
SELECT（无事务）从库负载均衡分发到多个从库
INSERT / UPDATE / DELETE主库写操作强制主库
事务内的 SELECT主库事务内读写一致性保证
SELECT FOR UPDATE主库悲观锁必须走主库
Hint 强制主库主库见下方代码

4.2 强制主库场景（Hint 用法）

以下场景必须强制走主库，防止主从延迟导致读到旧数据：

Java · ShardingSphere HintManager
// 场景：下单后立即查询订单详情（防主从延迟读到旧数据）
try (HintManager hintManager = HintManager.getInstance()) {
    hintManager.setReadReplicaRouteOnly();  // SS 5.x API
    // 或旧版：hintManager.setMasterRouteOnly();

    // 此次查询强制走主库
    Order order = orderMapper.selectByOrderId(orderId);
}

// 典型强制主库场景：
// 1. 支付成功后查询订单状态
// 2. 修改个人信息后立即回显
// 3. 库存操作后查库存余量
// 4. 对账/核查类业务

4.3 读写分离常见陷阱

🚨 高危陷阱：在 @Transactional 事务内，ShardingSphere 会将所有 SELECT 路由到主库。如果某个 Service 方法事务范围过大（包含大量只读查询），会导致主库压力大幅增加，与读写分离的初衷背道而驰。
Java · 反模式 vs 正确模式
// ❌ 反模式：事务范围过大，只读查询打到主库
@Transactional
public OrderDTO createOrderBad(OrderRequest req) {
    // 以下查询全部走主库！
    User user = userMapper.selectById(req.getUserId());
    List<Coupon> coupons = couponMapper.selectValid(req.getUserId());
    Address addr = addressMapper.selectDefault(req.getUserId());
    // ... 真正的写操作
    orderMapper.insert(order);
}

// ✅ 正确：将只读查询移到事务外
public OrderDTO createOrderGood(OrderRequest req) {
    // 事务外查询走从库
    User user = userMapper.selectById(req.getUserId());
    List<Coupon> coupons = couponMapper.selectValid(req.getUserId());
    Address addr = addressMapper.selectDefault(req.getUserId());

    doCreateOrder(req, user, coupons, addr); // 仅写操作在事务内
}

@Transactional
private void doCreateOrder(...) {
    orderMapper.insert(order);  // 走主库
}

5.1 应用层并行查询（移植 MPP 思路）

Java · 并行聚合
// 模拟DRDS跨分片并行查询：对多个分库并行发起查询后聚合
List<CompletableFuture<List<Order>>> futures = shards.stream()
    .map(shard -> CompletableFuture.supplyAsync(
        () -> orderMapper.queryByShard(shard, userId, startTime),

    .collect(Collectors.toList());

    .map(CompletableFuture::join)
    .flatMap(Collection::stream)

    .collect(Collectors.toList());

避坑清单
大促前 SQL Review 必查项（来自真实事故复盘）
  • SELECT * 禁止上大促：全字段查询破坏覆盖索引，增加网络传输和内存压力。必须指定业务需要的字段。
  • JOIN 超过 3 张表要预警：大促场景下复杂 JOIN 极易成为瓶颈，考虑拆分为多次单表查询或冗余字段。
  • UPDATE/DELETE 不加 WHERE 的裸操作严禁上线：必须有 Code Review 卡口和 SQL 审计工具（如 Yearning）拦截。
  • 大事务必须拆分：单个事务操作超过 1000 行或持续超过 3s，极易造成锁等待链，大促必死。
  • count(*) 不要在大表上实时查：超过千万级的表，count(*) 需要走缓存或预计算，不能每次请求都跑。
  • IN 列表不要超过 1000 个值：MySQL 对超大 IN 列表的优化有限，考虑改写为临时表 JOIN 或分批查询。
  • 索引数量不是越多越好：单表索引超过 5 个需要评估，写操作（INSERT/UPDATE）会维护所有索引，大促写入量大时代价很高。
  • 主从延迟不能假设为 0：大促写入量激增时，主从延迟可能达到秒级，下单后读从库的场景必须加 Hint 强制主库或加延迟重试。
  • 数值类型字段不要存字符串：隐式类型转换是索引失效的重灾区，统一在 DDL Review 阶段卡住。
  • 定时任务 SQL 要错峰：对账、数据同步等定时任务的 SQL 若在大促期间 0 点触发，会与业务流量争抢资源，必须调整到凌晨 3-4 点。
  • 慢 SQL 上线前必须 EXPLAIN：建立开发规范：涉及百万级表的 SQL，开发必须提供 EXPLAIN 结果作为 PR 附件。
  • UPDATE 不要用非索引字段做 WHERE：如 UPDATE orders SET status=1 WHERE third_party_no='xxx'，若 third_party_no 无索引则全表扫描加锁。
